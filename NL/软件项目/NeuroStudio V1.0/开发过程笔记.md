QT 提供 LGPL 和 GPL 两种开源协议 有什么不同

QT 提供的 LGPL（GNU Lesser General Public License， Lesser GPL）和 GPL（GNU General Public License）是两种不同的开源协议，核心区别体现在对衍生作品的开源要求和商业使用限制上，具体差异如下：
开源传染性不同
GPL 协议具有强传染性：如果你的程序使用了 GPL 许可的 QT 模块，那么你的整个程序（包括所有修改和衍生代码）都必须遵循 GPL 协议开源，且必须向用户提供完整的源代码，不允许闭源商用。
LGPL 协议传染性较弱：仅要求与 LGPL 模块直接链接的部分（通常是 QT 库本身）保持开源，而你的应用程序主体可以闭源商用。但需满足：通过动态链接（而非静态链接）使用 QT 库；允许用户替换或修改 QT 库文件；保留 QT 的版权声明和许可信息。
商业使用限制不同
GPL 协议：不允许闭源商用，任何基于 GPL 模块的商业软件必须完全开源，用户可自由复制、修改和分发。
LGPL 协议：允许闭源商用，无需公开应用程序的源代码，但需遵守动态链接、允许库替换等协议要求。
适用场景不同
GPL 协议适合完全开源的项目，或开发者愿意公开全部代码的场景。
LGPL 协议更适合希望闭源商用、同时利用开源 QT 库的商业项目，平衡了开源共享和商业利益。

整体为 前端 GUI (Qt/C++) 与 后端 Python (MNE) 独立进程通信。通信分为两类：控制消息（JSON/MessagePack） 与 大数据传输（共享内存 / memory-mapped files）。

•        数据通道（大数组）— POSIX shared memory (shm_open + mmap) 或 memfd / Windows Named Shared Memory；Python 端使用 multiprocessing.shared_memory 或 numpy.memmap。
•        同步/通知 — 控制通道可传共享内存标识与元数据；或使用事件fd / semaphore。
•        可选：后端提供 WebSocket/HTTP 服务以便调试或远程控制。
