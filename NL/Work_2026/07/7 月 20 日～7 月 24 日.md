### 2. 合并 / 澄清 `EEGManager` 与 `DataManagerController`

问题： 两者职责重叠：

|类|现状|
|---|---|
|`EEGManager`|Study/Subject/Session CRUD、当前 Session 指针、路径工具|
|`DataManagerController`|再包一层 CRUD + 日志 + 硬件检查 + EventBus 信号|

Controller 几乎总是 `EEGManager::getInstance()->xxx()`，形成「双层门面」。

建议：

- 方案 A（轻量）： Controller 专注「用例编排」（打开数据、进入采集、删除前检查），Repository 只管领域 CRUD
- 方案 B（中期）： 将 `EEGManager` 重命名为 `ProjectRepository`，Controller 注入 Repository 而非 singleton

同时 `getCurrentSession()` / `getCurrentAcquireSession()` 这种全局指针，应收敛到 `WorkspaceState` 或 `SessionContext`，避免 ConfigWidget、ContentWidget、Hardware 各自读 singleton。

### 3. 继续拆分「上帝 Widget」

`ConfigWidget`（~665 行） 仍集中：Tab 管理、采集控制、设备配置、阻抗、Montage、导出、Marker、插件 Tab……

`ContentWidget` 混合：历史波形 Tab、实时采集页、StackedWidget 切换、Pipeline 更新。

`EEGDevice.cpp`（~1500 行） 混合：串口、协议、CRC、双通道、阻抗、设备信息。

建议：

|组件|拆分方向|
|---|---|
|ConfigWidget|`AcquisitionPanel` + `DataProcessPanel` + `ConfigTabCoordinator`（已有 stackedIndex 映射可迁入）|
|ContentWidget|`HistoryContentPresenter` + `RealtimeContentPresenter`|
|EEGDevice|`GlProtocolCodec` + `SerialTransport` + `DeviceCommandSender`|

这与你们已在 DataManager 上做的 MVP 方向一致，应横向推广。

### 4. 统一菜单 / 命令体系（清理死代码）

存在三套机制：

- `MenuCommands.h` — 实际在用的右键菜单（`MenuFactory` + `canExecute`）
- `menu_config.json` + `MenuConfig` — 未被任何地方 load/引用，属于死代码
- `MenuFactory::getCommandsForType` — 定义了但 `createMenu` 并未按 type 过滤，而是遍历全部 command

建议：

- 删除或接入 `MenuConfig`：要么 JSON 驱动菜单，要么删掉 JSON 和 `MenuConfig`

- 命令注册从 Presenter 挪到 `ApplicationCommandRegistry`（MainWindow 启动时一次性注册）

## 三、中优先级

### 5. 强化 `AppEventBus` 或明确其边界

当前 EventBus 只是 Controller 信号的转发壳：

void AppEventBus::wireController(DataManagerController* controller)

{

connect(controller, &DataManagerController::dataSetOpened, this, &AppEventBus::dataSetOpened);

// ...

}

问题：

- 跨模块通信仍大量直接 `connect(具体类, 具体类)`
- EventBus 事件集很小，插件/ Survey 还要单独 listen

建议（二选一）：

- 扩展 EventBus： 所有 Session 生命周期、硬件连接、附件变更都走 Bus；Widget 只订阅 Bus
- 或收缩 EventBus： 删掉这层，Controller 信号由 `EEGWorkspaceCoordinator` 统一 wiring（Coordinator 模式做全）

无论哪种，目标都是：MainWindow 里不要散落 connect。

### 6. 完善插件架构

`IApplicationPlugin` 已有 Tab 扩展点，但：

- 插件仍是编译期静态链接（`registerBuiltInPlugins`）
- `AppContext` 暴露具体 Widget 指针，插件与 UI 耦合仍强
- `IPlatformHost` 能力偏少（打开数据、ensureSession），缺少：附件 API、日志、配置、硬件状态

建议：

- 扩展 `IPlatformHost`：`importAttachment`、`logOperation`、`subscribe(AppEvent)`
- 插件 Tab 注册改为 接口 + 工厂，不直接持有 `ConfigWidget*`
- 长期可考虑 Survey/Cloud 真正动态库 + 版本化 ABI

---

### 7. Session 领域模型瘦身

`Session` 同时承担：

- 文件系统对象（`m_path`、`Session.ini`）
- 运行时 `EegData` + `PipelineEngine`
- 实时采集状态（`SessionType` / `SessionState`）
- 附件集合

建议：

SessionMetadata — 磁盘、ini、附件列表

SessionRuntime — EegData、Pipeline（仅打开/采集时存在）

AcquisitionBinding — 实时 Session 与 Hardware 的绑定（已在 EegAcquisitionSession，可再强化）

`Session::m_path` 现为 public，多处直接拼接路径，应改为 `session->attachmentDir()` 等封装方法，避免路径规则散落。

### 8. CMake / 模块边界

问题：

- `NeuroStudioHost` INTERFACE 塞了 15+ include 路径，等于「全局 using namespace」
- `EEGApp` 的 `CMakeLists.txt` 手工列举数百个源文件
- Python 路径硬编码 `D:/SoftWare/Python`

建议：

- 每个子目录独立 `add_library(NeuroStudio.Views ...)` 并 `target_link_libraries` 声明依赖
- include 只暴露 public header，禁止跨模块 include 实现细节
- Python/VTK 路径走 `find_package` 或环境变量

## 四、长期 / 工程质量

### 9. Singleton 过多，可测试性差

项目中大量 `::instance()` / `getInstance()`：`EEGManager`、`DataManagerController`、`AppEventBus`、`DataTreeOperations`、各 Service、`SettingsFacade`、`HardwareContext`、`PythonInterpreter`……

建议：

- 启动时在 `main` / `ApplicationContext` 里构造依赖树，注入 Presenter/Controller
- 至少对 Interactor 层 支持构造函数注入，UI 层可暂时保留 singleton Facade

---

### 10. 缺少自动化测试

未发现 QTest / gtest 业务测试。架构演进风险高。

建议优先测：

- `DataTreeOperations::nodeObjects` / `itemPath`
- `DataManagerController` 打开/删除/重命名用例
- `stackedIndexForTab` 映射
- Hardware 协议解析（纯函数，可 offline 测）

---

### 11. 日志体系统一

现状：`qDebug` 散落、`AppLogger` 做操作审计、无 log category。

建议：

- 用 `QLoggingCategory`（`neuro.hardware`、`neuro.acquisition`、`neuro.data`）
- 发布版默认关闭 debug category
- 用户可见错误走 `AppLogger` + UI 提示，并返回结构化 `OpenResult` / `OperationResult`（不要统一「文件被占用」）
